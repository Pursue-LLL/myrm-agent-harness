## 一、设计原则

### 1.1 核心设计哲学

**减少可用工具数量 = 降低认知负载 = 提高决策准确性**

证据来源: [action_space.py](action_space.py), [tool_layers.py](tool_layers.py)

大语言模型在面对工具选择时，与人类面对选择困难症类似：
- 工具越多 → 动作空间越大 → 决策难度指数增长
- 工具越少 → 选项精简 → 准确率提升（零幻觉调用）

### 1.2 框架能力 vs 业务扩展（边界红线）

**术语**：对外 **「工具」= LLM 工具**（`ToolRegistry` 中的 `BaseTool`，CORE/COMMON/EXTENDED）。`toolkits/` 能力包、中间件、Skill 文档、编排状态机等**非 LLM 能力**禁止称作工具。见 `tool_catalog.py` · `TOOL_MANAGEMENT_SYSTEM.md`。

**Harness 是 Agent 框架，不是业务集成大杂烩。** 混淆层级会导致 toolkits 膨胀、定位模糊、维护成本失控。

| 层级 | 路径 | 放什么 | 禁止放什么 |
|------|------|--------|------------|
| 框架通用原语 | `toolkits/` | 与具体 SaaS 无关的可复用能力 | Google Calendar API 封装、HF Inference 专用工具 |
| 框架绑定元工具 | `agent/meta_tools/` | 需要会话/Planner/HITL 的工具包装 | 某一家 Notion/Linear 的硬编码 REST |
| 业务工作流 | `assets/prebuilt_skills/` | SKILL.md + contract + allowed-tools | 新的 `@tool()` 注册 |
| 三方集成（产品） | `app/integrations/`、MCP、channels | OAuth、渠道 SDK | 搬进 harness toolkits |
| 产品 REST | `app/api/` + `services/` | HTTP 域模型 | 默认 GeneralAgent 工具 |

**禁止模式**：单一厂商 SaaS 不得作为 harness toolkit；无 OAuth 产品化的 prebuilt skill 不得上架。

**正确替代**：
- 日历类 → 用户 MCP / 未来 `integrations/oauth` + skill；看板用现有 `kanban_*` 工具
- RSS/博客监控 → `blog-monitoring` skill + `web_search_tool` + `web_fetch_tool`
- HF 推理 → `ImageGenerationTools`（LLM 适配层）或用户 MCP

新增 harness 工具前自问：**去掉 Myrm 品牌后，别的 Agent 框架是否也会需要这个模块？** 若否 → skill 或 MCP。

**关键指标: ASCS (Action Space Complexity Score)**  
科学量化工具集的认知负担，指导用户优化工具配置。

## 二、工具层级体系 (Tool Layer System)

### 2.1 三层架构设计

证据来源: [tool_layers.py:23-36](tool_layers.py)

```python
class ToolLayer(IntEnum):
    CORE = 1      # 核心工具，始终存在，不可关闭
    COMMON = 2    # 通用工具，默认存在，前端可控制开关
    EXTENDED = 3  # 扩展工具，按需加载或定义始终发送
```

### 2.2 分层设计目标

**目标**: 最大化 Prompt Cache 命中率

**层级说明**:

#### 2.2.1 CORE 层 (Layer 1)

- **特征**: 通用 Agent 基线，Turn1 eager；前端无开关（Fast 模式由 converter 关闭 file/bash）
- **工具数量**: **7**（`tool_layers.py:57-63`）
- **Token 消耗**: ~4,060 tokens（2026-07-03 实测，见 `DEFAULT_AGENT_TOKEN_INVENTORY.md` §二）

| 工具名 | 常驻理由 |
|--------|----------|
| web_fetch_tool, bash_code_execute_tool, file_read/write/edit_tool, glob_tool, grep_tool | 通用工作助手基线（非 IDE 专用；file/bash 为沙箱内日常操作） |

**设计原则**: 仅包含 Agent 基线工具；详细 token 逐项见 `DEFAULT_AGENT_TOKEN_INVENTORY.md` §二。

#### 2.2.2 COMMON 层 (Layer 2)

- **特征**: 默认开启，GUI 可关；组内 memory 优先于 web_search（`get_tool_registry_sort_key`）
- **工具数量**: 注册 **5** 个
- **默认 Turn1 bind**: memory×3 + web_search（见 `DEFAULT_AGENT_TOKEN_INVENTORY.md` §三）

| 工具名 | 加载条件 |
|--------|----------|
| memory_recall/save/manage_tool | enable_memory |
| web_search_tool | GUI 可关 |
| todo_write | opt-in（planning） |

#### 2.2.3 EXTENDED 层 (Layer 3)
- **特征**: 按需 Turn1 或 RUNTIME_ONLY 绑定
- **缓存**: 放最后使其变化不影响 CORE/COMMON 的缓存
- **工具数量**: 登记 **55** EXTENDED（harness 静态 + server bootstrap 5；见 TOOL_COUNT 块）
- **Token 消耗**: 典型 ~2,246 tokens，满载 ~7,290+ tokens

**子分类**:

| 子类 | 加载条件 | 典型工具 | Token 消耗 |
|------|---------|---------|----------:|
| 能力发现网关（**TURN1**，DeferEconomics 判定） | searchable skills / 大 defer 池 | discover_capability_tool | ~819（条件绑定） |
| bash_process_tool | bash 启用时 Turn1 | bash_process_tool | ~200 |
| 记忆工具（COMMON） | 启用记忆系统 | memory_recall_tool, memory_save_tool, memory_manage_tool | ~840 |
| 会话搜索（EXTENDED，opt-in） | `memoryEnableConversationSearch=true` 且非无痕 | conversation_search_tool | ~67 |
| 技能工具（Turn1） | 有技能后端 | skill_select_tool, skill_manage_tool | ~343 |
| 浏览器工具 | `enabled_builtin_tools: browser` | browser_navigate_tool, browser_snapshot_tool, ... (8个) | ~535 |
| 定时任务工具 | cron 能力 wired；勾选 cron 时 Turn1 | cron_manage_tool | ~827 |
| Wiki 工具 | `enabled_builtin_tools: wiki` | wiki_query_tool, wiki_ingest_tool (2个) | ~95 |
| 统一知识检索 | wiki + memory 双开 | knowledge_recall_tool (1个) | ~55 |
| 子 Agent 工具 | SubagentManagementExtension + entitlements | delegate_task_tool, batch_delegate_tasks_tool, send_teammate_message_tool, steer_subagent_tool, ... (7个) | ~411 |
| UI 渲染工具 | `enabled_builtin_tools: render_ui` | render_ui_tool | 223 |

| 看板工具 | `enabled_builtin_tools: kanban` | worker 5 + orchestrator 7（`kanban_link` 合并依赖）；board CRUD 走 REST/GUI | -
| Server 业务工具 | skill/凭证/notify 配置 | x_search_tool, knowledge_recall_tool, channel_notify_tool, image_tool, video_tool, tts_generate (6个) | 按凭证 |

### 2.3 排序规则

证据来源: [tool_layers.py:47-48](tool_layers.py)

**排序算法**: 按层级排序，同层级内按名称字母序

**缓存原理**: Prompt Cache 是前缀匹配，CORE 工具放最前面可保证永远被缓存

**实际效果**: 
- 固定部分 ~7,300+ tokens 全部被缓存（仅收 5-10% 费用）
- 首轮后续调用大幅节省成本

## 三、核心/外围技能机制 (Core/Peripheral Skills)

### 3.1 设计目标

**差异化认知负载管理**

证据来源: [myrm-agent-server/app/api/agents/agent.py:644-651](../../../myrm-agent-server/app/api/agents/agent.py)

### 3.2 核心技能 (Core Skills)
- **特征**: 常驻在模型 Prompt 中
- **认知负载**: 100%
- **适用场景**: 高频使用的技能

### 3.3 外围技能 (Peripheral Skills)
- **特征**: 按需加载，只在用户明确说需要时才注入
- **认知负载**: 50%（常驻负载减半）
- **适用场景**: 低频使用的技能

### 3.4 成本计算公式

```python
cost = ActionSpaceProfiler.BASE_TOOL_COST + (len(skill.description or "") // 50)
if not is_core:
    cost = int(cost * 0.5)  # 外围工具因按需加载，常驻认知负载减半
```

证据来源: [myrm-agent-server/app/api/agents/agent.py:648-650](../../../myrm-agent-server/app/api/agents/agent.py)

## 四、动作空间复杂度评估 (ASCS)

### 4.1 核心引擎: ActionSpaceProfiler

证据来源: [action_space.py:17-108](action_space.py)

**ASCS (Action Space Complexity Score)**: 量化大模型的动作空间复杂度

### 4.2 计算策略

#### 4.2.1 基础成本

| 成本类型 | 数值 | 说明 |
|---------|-----:|------|
| BASE_TOOL_COST | 10 | 每个工具的存在都会分散 LLM 的注意力 |
| PARAM_COST | 5 | 每个参数的复杂度 |
| NESTING_COST | 10 | 嵌套对象的额外惩罚 |

#### 4.2.2 外部工具估算

| 工具类型 | 估算成本 | 说明 |
|---------|--------:|------|
| MCP 工具 | 400 | 外部服务往往包含多个具备极深嵌套对象的工具 |
| 内置工具 | 100 | 系统内置的 web_search, browser 等 |

#### 4.2.3 技能工具计算

```python
# 启发式估算
cost = BASE_TOOL_COST + (len(skill.description) // 50)

# 核心/外围差异化
if not is_core:
    cost = int(cost * 0.5)
```

### 4.3 准确度计算

```python
max_safe_score = 1500
noise_level = min(100, round((total_score / max_safe_score) * 100))
accuracy_level = 100 - noise_level
```

证据来源: [myrm-agent-server/app/api/agents/agent.py:660-662](../../../myrm-agent-server/app/api/agents/agent.py)

### 4.4 警告阈值

| 等级 | 阈值 | 准确度 | 提示 |
|------|-----:|-------:|------|
| 绿色 ✨ | noise_level ≤ 50 | > 50% | "极佳决策准确率：动作空间极度精简，Agent 能发挥最高级别的零幻觉调用" |
| 黄色 ⚖️ | 50 < noise_level < 80 | 20-50% | "能力与准确率平衡：工具较多，部分相似功能可能导致模型混淆" |
| 红色 ⚠️ | noise_level ≥ 80 | < 20% | "准确率暴跌风险：工具过度泛滥！模型面临极严重的幻觉风险，请清理" |

## 五、智能管家提醒系统 (Smart Pruning)

### 5.1 闲置技能检测

证据来源: [myrm-agent-frontend/src/components/ui/chat-window/agent-config-panel/AgentConfigEditDialog.tsx:327-334](../../../myrm-agent-frontend/src/components/ui/chat-window/agent-config-panel/AgentConfigEditDialog.tsx)

**检测逻辑**:
```typescript
const staleCoreSkills = useMemo(() => {
  return localSkillIds.filter((id) => {
    const isCore = localSkillConfigs?.[id]?.is_core ?? true;
    if (!isCore) return false;
    const skill = enabledSkills?.find((s) => s.id === id);
    return skill?.usage_stats?.lifecycle_status === 'stale';
  });
}, [localSkillIds, localSkillConfigs, enabledSkills]);
```

**判定标准**: 30 天未调用的核心技能 → `lifecycle_status = 'stale'`

### 5.2 一键净化功能

**智能降级**: 自动将闲置的核心技能降级为外围技能

**效果**: 
- 释放 50% 认知负载
- 提升准确度
- 技能仍可用，只是变为按需加载

## 八、MCP 工具特殊处理

### 8.1 强制路由到 EXTENDED 层

证据来源: [tool_layers.py:102-116](tool_layers.py)

**设计原则**: 保护大模型 Prompt Prefix Cache

```python
def get_tool_layer(tool_name: str) -> ToolLayer:
    """获取工具层级
    
    [核心架构约束]:保护大模型 Prompt Prefix Cache
    所有动态挂载的外部工具(特别是 MCP 提供的工具,如 github_search 等)
    必须强制路由到 EXTENDED 层级(Prompt尾部).
    这能确保前部巨大的 System Prompt 和核心通用工具不被打乱,缓存命中率锁定在100%.
    
    Returns:
        工具层级,未注册的工具(含所有 MCP 工具)默认为 EXTENDED(放在最后)
    """
    return _TOOL_LAYERS.get(tool_name, ToolLayer.EXTENDED)
```

### 8.2 高成本估算

**MCP 工具成本 = 400 tokens**

原因: 外部服务往往包含多个具备极深嵌套对象的工具，保守估算值确保准确性

## 十一、已实现的优化

### 11.1 使用数据分析 ✅ 已实现

**实现**: `lifecycle_status = 'stale'` (30天未调用自动标记) + `curator engine` (生命周期治理)

**效果**:
- 前端智能管家自动检测闲置技能
- 一键净化：核心(100%负载) → 外围(50%负载)
- 完整的使用统计 (`usage_stats`)

### 11.2 自适应工具加载 ✅ 已实现

**实现**: `discover_capability_tool` (BM25+Embedding) — indexes external skills for unified search

**效果**:
- DeferEconomics：小 defer 池 + 无外部技能 → 不绑 discover 网关（省 ~238 tok/Turn1）
- Agent 通过 discover 命中外部技能 → `skill_select_tool` 加载 SOP 后执行
- 严禁 append 非 Turn1 schema 到 bind_tools（框架 11.1 prefix cache 底线）

### 11.3 工具推荐系统 ✅ 已实现

**实现**: `discover_capability_tool` 统一搜索 Native Discoverable Tools + External Skills

**效果**:
- BM25 + Embedding 混合语义搜索
- 跨语言查询支持 ("火车票/railway ticket/train booking")
- 搜索命中后必须用 `skill_select_tool` 加载 SOP

### 11.4 动态 Schema 修正 (Dynamic Schema Modification) ✅ 已实现

**实现**: `dynamic_schema_modifier` 钩子 (在 `ToolRegistry.resolve()` 中调用) + `with_dynamic_hints` 装饰器

**效果**:
- 工具可以在运行时根据当前会话中可用的其他工具动态调整自己的 `description`。
- 避免模型因为工具描述中提及了当前不可用的工具而产生幻觉调用。
- 保护 Prompt Cache：通过在解析阶段动态修正并生成不可变副本，确保工具描述与实际能力严格一致。

## 十三、核心价值主张

**工具设计策略的核心价值**:

1. **科学量化**: 从主观判断到客观指标
2. **实时反馈**: 从事后分析到事前预防
3. **智能化**: 从手动配置到自动优化
